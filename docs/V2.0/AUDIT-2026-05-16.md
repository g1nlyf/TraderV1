## Critical Gaps (Verified by Test, Not Documentation)

### GAP-1: Data Pipeline Not Connected — BLOCKING EVERYTHING

**Finding:** 741,633 pool_transactions exist in legacy DB. Stage 2 `raw_source_events` has 60 rows — all from calibration quote observations or acceptance fixtures. Zero real token discovery or wallet transaction data flows into Stage 2.

**Root cause:** `write_raw_source_event` and the legacy ingestion mappers exist and are correct, but they are **only called in acceptance fixture code**. The live scheduler (`WalletScarperScheduler`), discovery service, and transaction parser do NOT call them. There is no bridge between the legacy pipeline and Stage 2.

**Impact:** Every V2 tool that depends on real data returns empty results:
- `token.scan_universe` → `token_candidates_created: 0`
- `token.request_deep_parse` → corpus with `no_trade_events_available` flag
- `wallet.extract_from_token` → 0 wallet candidates from real tokens
- `wallet.profile_history` → only works on fixture wallets
- `wallet.list_elite` → only 1 fixture wallet

**Fix required:** Modify the discovery service and/or scheduler to call `write_raw_source_event` on every new token discovery or pool transaction event. The mappers are ready; they just need to be wired in.

### GAP-2: Tracked Wallet Signals Not Connected to Stage 2

**Finding:** 38 tracked wallets in `tracked_wallets` (legacy). `tracked_wallet_signal_events` in Stage 2 has only fixture rows. The legacy `LiveMonitor` checks `pool_transactions` but does not call `record_tracked_wallet_signal_event`.

**Impact:** Hermes never receives real wallet buy/sell signals. The `agent.record_trading_decision` chain never fires on real data.

**Fix required:** LiveMonitor (or a new Stage 2 wallet monitor) must call the `wallet.record_signal_event` tool when a tracked wallet transacts.

### GAP-3: No Continuous Worker Daemon

**Finding:** `WorkerPoolService` exists in `stage2/workers/service.py` but there is no CLI command to start a continuous V2/Stage 2 worker loop. The legacy `run` command starts `WalletScarperScheduler` (APScheduler) but that scheduler does not include V2 token scanning, wallet profiling, or Hermes decision loops.

**Impact:** The system cannot run 24/7 autonomously. Every V2 operation requires manual CLI invocation.

### GAP-4: No Adaptive Market Loop (Sprint 3 Not Started)

**Finding:** `active_token_sessions` table exists in schema but `ActiveTokenSession` lifecycle service is not implemented. No cadence policy. No priority queue for open positions vs research.

**Impact:** Sprint 3 acceptance gates are all unmet.

### GAP-5: Shadow Readiness Remains Gap-Blocked

**Finding:** Acceptance returns `shadow_status: gap_report_required`. The `live_data_acceptance_windows` only have calibration smoke fixtures, not real multi-source quote observations with route quality and fill comparison evidence.

**Impact:** Cannot graduate to shadow mode.

### GAP-6: token.scan_universe Quality Flag Issue

**Finding:** Even the 60 real calibration quote events in Stage 2 fail normalization with `missing_external_id` and `low_source_confidence`. These are quote-only observations (no token discovery metadata), so the normalizer correctly rejects them as token candidates. This is correct behavior, not a bug — but it means calibration events will never produce token candidates.

**Fix:** Token discovery data (from DexScreener pairs API, GeckoTerminal pools API) needs a separate ingestion path that maps pool discovery payloads (not quote-only calibration payloads) into `raw_source_events` with `source_type: token_discovery`.

## What Needs to be Done — Prioritized Work Plan

### Priority 1: Wire Legacy Data Into Stage 2 (Unblocks Everything)

**File:** `walletscarper/services/discovery.py` (and/or `scheduler.py`)

In `DiscoveryService.discover_new_tokens()` (or wherever DexScreener/GeckoTerminal results are processed), add:

```python
from walletscarper.stage2.legacy_ingestion import map_dexscreener_payload, write_raw_source_event
from walletscarper.stage2.events import RawSourceEventLog
# For each discovered token/pool pair:
draft = map_dexscreener_payload(raw_pair_dict)
await write_raw_source_event(draft, raw_log)
```

Similarly for pool_transactions → wallet trades ingestion:

```python
from walletscarper.stage2.legacy_ingestion import map_bitquery_raw_trade, map_dexpaprika_payload
```

This single change makes `token.scan_universe` return real candidates.

## Summary

The V2.0 system architecture and services are correctly implemented and verified working in fixture mode. The orchestration decision path (Signal → Risk → Paper → Exit → Review → Memory) passes end-to-end. All 24 Hermes tools are real implementations.

The system fails to deliver V2.0 value because the data pipeline bridge is missing: 741,633 legacy transactions and 38 tracked wallets are not connected to the Stage 2 event bus. The legacy ingestion mappers exist and are correct — they just need to be called from the live pipeline.

This is not a fundamental architectural problem. It is a wiring gap that requires targeted changes to 2-3 files, not a rewrite.
